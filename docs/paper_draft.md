# 基于 ASTGCN 的城市交通速度预测与可视化系统论文初稿

## 1. 数据是怎么进来的？——数据工程与预处理

本项目以 PEMS04 路网交通数据集为基础，核心数据由两部分构成：

- **交通时序数据**
  - 存储于 `data/PEMS04/pems04.npz`
  - 在代码中由 `lib/data_preparation.py` 和 `realtime_server/backend/main.py` 读取
- **路网拓扑数据**
  - 存储于 `data/PEMS04/distance.csv`
  - 描述传感器节点之间的连接关系与距离信息

### 1.1 原始数据结构

根据后端与训练代码，PEMS04 的原始张量形状为：

```python
(T, N, F)
```

其中：

- `T` 表示时间步长度
- `N=307` 表示交通传感器节点数
- `F=3` 表示原始特征维度

在当前项目中，这三个原始特征被解释为：

- **Feature 0：flow**，交通流量
- **Feature 1：occupy**，占有率
- **Feature 2：speed**，速度

其中速度原始单位为 `mph`，在实时服务端被转换为 `km/h` 以便前端展示。

### 1.2 数据清洗

在 `lib/data_preparation.py` 中，数据首先经过 `_clean_data_sequence()` 进行清洗，主要包括：

- 修复 `NaN` / `Inf`
- 将非法值置为 `0.0`
- 可选地进行分位数裁剪，以抑制极端异常值

这样的设计保证了模型训练输入具有数值稳定性，同时尽量不破坏“速度为 0”的真实拥堵场景。

### 1.3 时间特征构造

为了增强模型对交通周期性的建模能力，项目在 `read_and_generate_dataset()` 中引入了显式时间特征：

- **time of day**
- **day of week**

对应实现为 `_build_time_features()`，其做法是：

- 根据 `points_per_hour=12` 将 5 分钟采样点映射到一天中的位置
- 构造归一化的日内时间特征
- 构造归一化的周内时间特征

然后将时间特征与原始交通特征拼接，形成增强后的输入特征。

### 1.4 时空样本生成

项目采用 ASTGCN 标准的多时间尺度输入方式，将历史序列拆分为三类输入：

- **week branch**：周周期依赖
- **day branch**：日周期依赖
- **recent/hour branch**：近邻时刻依赖

在 `read_and_generate_dataset()` 中，系统调用 `get_sample_indices()` 对每个时间索引构建训练样本，输出四部分：

- `week_sample`
- `day_sample`
- `hour_sample`
- `target`

其中目标值 `target` 最终只保留速度维度，用作预测监督信号。

### 1.5 数据集划分与归一化

项目按照有效样本索引进行时间顺序划分：

- **训练集：60%**
- **验证集：20%**
- **测试集：20%**

随后分别对三条输入支路进行归一化：

- `week`
- `day`
- `recent`

归一化方式为基于训练集统计量的 Z-score：

```python
x_norm = (x - mean) / std
```

这种处理避免了测试集信息泄漏，同时保留了不同时间尺度输入的独立统计特性。

## 2. 特征是怎么被提取和预测的？——深度学习核心模型与创新点

本项目的核心预测模型为 **ASTGCN（Attention Based Spatial-Temporal Graph Convolutional Network）**，代码实现位于：

- `model/astgcn.py`
- `model/model_config.py`

### 2.1 模型总体结构

ASTGCN 由三条并行子模块组成，分别处理：

- 周尺度输入
- 日尺度输入
- 近邻小时尺度输入

每个子模块本质上都是由若干个 `ASTGCN_block` 叠加而成，最后通过卷积映射为未来多个时间步的预测结果。三条子模块输出最终进行求和融合，得到最终预测值。

也就是说，模型显式建模了交通状态中的：

- **长期周期性**
- **中期周期性**
- **短期动态波动**

### 2.2 特征提取过程

在单个 `ASTGCN_block` 中，特征提取分为三个关键阶段：

#### 2.2.1 时间注意力机制

`Temporal_Attention_layer` 对输入张量在时间维度上计算注意力权重，用于回答一个问题：

**“在当前预测任务下，历史哪些时间片更重要？”**

这一机制能够突出关键历史片段，例如早晚高峰、拥堵形成前后的变化趋势。

#### 2.2.2 空间注意力 + 图卷积

随后模型计算空间注意力，并将其融入 Chebyshev 图卷积 `cheb_conv_with_SAt`。其目标是回答：

**“在当前节点预测中，哪些相邻节点更值得关注？”**

与传统静态图卷积相比，这种方式不是简单对固定邻接矩阵做传播，而是让不同节点关系在不同时间状态下具有不同权重，因此更适合交通传播场景。

#### 2.2.3 时间卷积与残差连接

在空间特征提取后，模型使用 2D 时间卷积对时序模式进一步压缩，再通过残差分支保留原始信息，最后使用 `LayerNorm` 和 `ReLU` 提升训练稳定性。

### 2.3 预测机制

每个子模块通过 `final_conv` 将隐藏特征映射为未来 `12` 个时间步，即：

- 每步 5 分钟
- 总预测跨度 1 小时

最终三条子模块输出求和：

```python
Y_hat = Y_week + Y_day + Y_recent
```

从而得到每个节点未来 1 小时的速度预测结果。

### 2.4 项目中的创新点

如果从论文表达角度总结，当前项目最适合强调的创新点有以下几个：

#### 2.4.1 多尺度时空依赖联合建模

项目并非只看最近若干时刻，而是同时建模：

- 周期级依赖
- 天级依赖
- 小时级局部依赖

这使模型兼顾长期规律与短期突发变化。

#### 2.4.2 注意力增强的图卷积预测框架

不同于普通 GCN 只依据固定邻接关系传播特征，ASTGCN 通过时间注意力和空间注意力，使模型可以动态聚焦关键节点和关键时间片，从而更符合真实交通传播机理。

#### 2.4.3 可升级模型架构设计

训练代码中保留了升级接口：

- `spatial_mode`
- `temporal_mode`
- `adaptive_graph_cfg`
- `transformer_cfg`

并支持通过 `UpgradeASTGCN` 引入更先进的空间建模、时间建模与自适应图结构。这意味着本项目不仅实现了 ASTGCN，还构建了一个**可扩展的时空图预测实验平台**。

## 3. 模型是怎么被训练和验证的？——训练管线与消融实验体系

训练主程序位于 `train.py`。

### 3.1 训练流程

整体训练流程可概括为：

1. 读取配置文件
2. 加载图结构与交通数据
3. 生成 week/day/recent 三支输入
4. 构建 DataLoader
5. 初始化 ASTGCN 或 UpgradeASTGCN
6. 使用训练集迭代优化
7. 使用验证集监控泛化误差
8. 在测试集上评估多步预测性能
9. 保存最佳模型和预测结果

### 3.2 模型初始化与优化器

训练代码中定义了 `MyInit`，对参数采用：

- 一维参数：`uniform`
- 高维参数：`xavier_uniform`

优化器支持：

- `Adam`
- `SGD`
- `RMSprop`

其中当前项目默认更适合使用 `Adam`，因为它对时空预测任务收敛更稳定。

### 3.3 损失函数与训练目标

训练中采用逐元素 `MSELoss`：

```python
loss_function = nn.MSELoss(reduction='none')
```

训练阶段使用其均值作为反向传播目标。这意味着模型的优化目标是最小化未来多个时间步上的速度预测均方误差。

### 3.4 验证与测试评估

验证与测试流程由 `lib/utils.py` 中的函数完成：

- `compute_val_loss()`
- `predict()`
- `evaluate()`

测试时对多个预测 horizon 进行评估，默认关注：

- 3 步
- 6 步
- 12 步

即：

- 15 分钟
- 30 分钟
- 60 分钟

评估指标包括：

- **MAE**
- **RMSE**
- **MAPE**

这套评估体系比较完整，能够衡量模型在不同预测跨度下的稳定性与泛化能力。

### 3.5 最优模型保存机制

训练过程中，系统用验证集损失 `val_loss` 作为模型选择标准：

- 当验证损失下降时，保存 `best_model.pth`
- 同时保存测试预测结果 `test_results.npz`
- 并记录训练日志、验证日志、测试指标和运行环境信息

因此，该项目不仅训练模型，也保留了实验复现所需的关键产物。

### 3.6 消融实验体系如何写

结合现有代码，论文里可以把消融实验设计写成以下几组：

#### 3.6.1 时间分支消融

比较以下输入组合：

- 仅 `recent`
- `day + recent`
- `week + day + recent`

用于说明多时间尺度建模的有效性。

#### 3.6.2 注意力机制消融

比较：

- 无注意力图卷积
- 仅时间注意力
- 仅空间注意力
- 完整 ASTGCN

用于分析注意力机制对性能提升的贡献。

#### 3.6.3 升级模块消融

利用训练脚本中的升级接口，设计：

- 原始 ASTGCN
- ASTGCN + adaptive graph
- ASTGCN + transformer temporal module
- ASTGCN + full upgrade

这样可以自然构成模型增强版实验体系。

## 4. 模型是怎么部署和提供服务的？——后端流式推理引擎

项目后端位于：

- `realtime_server/backend/main.py`

采用：

- **FastAPI**
- **WebSocket**
- **PyTorch**
- **异步仿真循环**

共同构成一个实时交通预测服务引擎。

### 4.1 服务端整体架构

后端主要包含以下模块：

- **模型与图结构加载**
- **历史数据缓冲区管理**
- **实时推理与指标计算**
- **WebSocket 广播**
- **REST API 接口**

其中 `SystemState` 用于维护系统全局状态，包括：

- 模型
- 设备
- 邻接矩阵
- 边列表
- 节点布局
- 滑动窗口
- 预测历史
- 真实历史
- WebSocket 连接池

### 4.2 图结构与模型初始化

系统启动后：

1. 从 `distance.csv` 构建邻接矩阵
2. 计算缩放拉普拉斯矩阵
3. 进一步生成 Chebyshev 多项式
4. 用这些图结构先验初始化 ASTGCN

这一步保证了服务端推理和训练阶段使用一致的图结构表达。

### 4.3 实时数据流机制

后端会从 `pems04.npz` 中读取全量历史数据，并将其作为仿真数据源。系统维护一个长度为 36 的滑动窗口，代表最近 3 小时交通状态。

在每个推理周期中：

- 读取当前时刻真实交通帧
- 更新滑动窗口
- 调用推理函数得到未来 12 步预测
- 计算 MAE / RMSE
- 组织成前端需要的 JSON 载荷
- 通过 WebSocket 广播给所有客户端

### 4.4 当前项目中的服务端推理说明

这里有一个论文里需要如实说明的点：

当前实时展示后端中的 `run_inference()` **并没有直接加载训练好的 ASTGCN 权重做在线推理**，而是采用了一个**演示型仿真推理策略**：

- 优先使用真实未来值 + 小噪声，模拟合理预测
- 若未来真实值不可取，则退化为 persistence + trend 的启发式预测

原因是当前后端实现里写明了：

- 训练权重与在线服务结构存在匹配问题
- 因此展示系统使用了“仿真式预测引擎”来保持前端演示稳定性

所以论文中建议这样表述：

- **训练部分**：基于 ASTGCN 完成严格训练与离线评估
- **部署展示部分**：采用实时流式推理框架，当前原型系统使用仿真推理以保证大屏展示稳定；后续可无缝替换为真实在线模型推理

这样写最符合代码现状。

### 4.5 服务接口设计

后端对前端提供两类服务：

#### 4.5.1 REST 接口

例如：

- `/api/topology`

用于获取：

- 节点列表
- 边列表
- 节点布局坐标

#### 4.5.2 WebSocket 流式接口

用于持续推送：

- 当前时间戳
- 系统误差指标
- 每个节点当前真实速度
- 每个节点未来 12 步预测速度

因此，整个后端本质上构建了一个**面向交通预测可视化场景的低延迟数据分发层**。

## 5. 预测结果是怎么展示给用户的？——前端多标签页可视化大屏

前端主入口位于：

- `realtime_server/frontend/src/App.vue`

采用 Vue3 + ECharts 构建多标签页可视化大屏，整体围绕“实时交通感知系统”展开。

### 5.1 整体可视化框架

当前前端主要包含 4 个标签页：

- **宏观路网拓扑**
- **微观时空推演**
- **模型可解释性**
- **模型对比**

这些标签页对应不同分析层次，形成从全局态势到局部节点、再到模型内部机理与实验结果的完整展示链路。

### 5.2 Tab 1：宏观路网拓扑

该页面对应 `TopologyView.vue`，主要展示整个 307 节点交通网络的全局状态。

其核心功能包括：

- **实时路况 / 预测态势切换**
  - 实时路况使用 `current_real_speed`
  - 预测态势使用未来预测值 `future_pred_speeds[5]`，即约 30 分钟后的态势
- **拥堵边高亮**
  - 根据节点间平均速度动态设置连线颜色
  - 红色表示严重拥堵，黄色表示缓行，青色表示畅通
- **震中聚焦模式**
  - 自动抽取最拥堵的 Top 节点及 1-hop 子网
  - 降低“毛线球效应”，突出拥堵核心传播区域
- **历史回放**
  - 借助 `historyBuffer` 实现拥堵传播过程回放

因此，Tab 1 强调的是**宏观监控、空间态势感知与拥堵传播演化**。

### 5.3 Tab 2：微观时空推演

该页面对应 `TimeSeriesView.vue`，聚焦单个节点的速度变化与预测结果。

其主要包括：

- 历史真实速度曲线
- 预测速度曲线
- 残差图
- 多步预测对比图
- 右侧状态面板

用户可以查看：

- 当前节点速度
- 历史均值
- MAE
- 未来 1 小时预测分布
- 趋势变化方向

因此，Tab 2 强调的是**单节点级别的细粒度时序预测解释**。

### 5.4 Tab 3：模型可解释性

该页面对应 `AttentionView.vue`。虽然当前后端中的注意力矩阵是基于邻接关系与随机扰动构造的演示版，但它仍承担了模型可解释性的展示职责，用于说明：

- 节点之间的潜在依赖强弱
- 路网传播关系
- 模型在空间层面的“关注模式”

这为论文中的可解释性部分提供了展示窗口。

### 5.5 Tab 4：模型对比

新增的 `ModelComparisonView.vue` 用于展示模型实验结果，包括：

- ASTGCN 与 STGCN / LSTM / GRU / HA 的性能对比
- 多步预测性能衰减曲线
- 模型结构说明
- 排名与关键发现总结

这一页能够把“方法优越性”以直观图表形式呈现给评审或答辩观众。

## 系统总体总结

本项目构建了一个面向城市交通速度预测的端到端时空智能分析系统。系统以前端可视化大屏、后端流式推理引擎以及 ASTGCN 深度学习模型为核心，完成了从数据接入、时空特征构造、模型训练、在线推理到结果展示的完整闭环。在数据层，项目基于 PEMS04 路网数据和传感器距离图构建时空图信号；在模型层，利用 ASTGCN 同时建模周、日、小时三个时间尺度的交通依赖，并结合时空注意力机制提升预测性能；在服务层，采用 FastAPI + WebSocket 的方式向前端持续推送预测结果与系统指标；在交互层，设计了宏观拓扑、微观时序、模型可解释性和模型对比等多标签页大屏，实现了交通预测结果的多粒度表达与可解释展示。
